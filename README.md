# Key-Value Storage on Flash (SSD/MMC Simulator)


## 📌 Описание

Реализация движка key-value хранилища на C, работающего поверх симулятора flash-памяти. Проект моделирует реальные ограничения flash-устройств: постраничная запись, поблочное стирание, `0xFF` как состояние стёртой ячейки. Записи хранятся в лог-структурированном append-only формате с проверкой целостности через CRC32.

Поиск ключа оптимизирован собственной хеш-таблицей.


## 🧱 Архитектура проекта

Пять слоёв, каждый с одной зоной ответственности:

```
main.c                    — демо / тестовый сценарий
  └── kvs                 — API хранилища (set / get / delete / sync)
        ├── hashmap       — кэш в памяти: O(1)-поиск по ключу
        └── hal           — аппаратная абстракция: лог-формат, CRC, GC
              └── flash   — сырое чтение/запись/стирание страниц flash.bin
```

---

### 1. Flash

Обёртка над бинарным файлом, моделирующая 1 МБ flash-чип:

- 1 МБ всего, страница — 512 байт, блок — 16 КБ (2048 страниц, 64 блока)
- `flash_read_page` / `flash_write_page` — постраничный ввод-вывод
- `flash_erase_block` — заполняет блок байтами `0xFF`
- При первом запуске создаёт и инициализирует бинарный файл

---

### 2. HAL (Hardware Abstraction Layer)

Лог-структурированное хранилище записей поверх flash:

- Каждая запись хранится в одной или нескольких страницах: `[заголовок | ключ | значение | CRC32]`
- Поля заголовка: `status` (EMPTY `0xFF` / VALID `0xFE` / OBSOLETE `0x00`), `pages_count`, `key_size`, `value_size`
- `hal_write` — вычисляет нужное количество страниц, дописывает запись начиная с `next_page`
- `hal_iterate` — последовательно проходит все VALID-записи с проверкой CRC и отдаёт их через callback (используется KVS при старте для заполнения кэша)
- `hal_delete` — помечает запись как OBSOLETE прямо на месте, без стирания блока
- При старте сканирует страницы для восстановления позиции `next_page` — данные переживают перезапуск
- Перед каждой записью проверяет наличие свободного места и при необходимости запускает GC

Wear leveling реализован неявно: append-only запись гарантирует равномерный обход всех блоков, без необходимости в явных счётчиках.

Сам HAL не хранит индекс по ключам и не оптимизирован для поиска — это вынесено на уровень KVS, как изначально и предполагала задача.

---

### 3. KVS

API хранилища с явным разделением кэша в памяти и физической записи на flash — по аналогии с page cache в Linux:

```c
kvs_t *kvs_create(const char *filename);
int    kvs_set(kvs_t *kvs, const char *key, const char *value);
int    kvs_get(kvs_t *kvs, const char *key, char *value, uint16_t *value_size);
int    kvs_delete(kvs_t *kvs, const char *key);
int    kvs_sync(kvs_t *kvs);
void   kvs_simulate_power_loss(kvs_t *kvs, size_t power_loss_after);
void   kvs_destroy(kvs_t *kvs);
```

- `kvs_set` / `kvs_delete` работают только с кэшем (структура hashmap), помечая запись как `DIRTY` или `DELETED` — на flash в этот момент ничего не пишется.
- `kvs_get` читает значение прямо из кэша — поиск по ключу выполняется за O(1) благодаря хеш-таблице вместо линейного перебора записей.
- `kvs_sync` — единственная точка физической записи: проходит по кэшу и для каждой `DIRTY`-записи делает `hal_delete` + `hal_write`, а для `DELETED` — `hal_delete` и убирает запись из кэша. Останавливается и возвращает ошибку при сбое записи (например, при потере питания).
- `kvs_create` восстанавливает кэш из flash через `hal_iterate`, поэтому после перезапуска данные и скорость поиска не теряются.

Длина ключа ограничена 255 байтами (`uint8_t`), значения — 65535 байтами (`uint16_t`).

---

### 4. Hashmap

Собственная реализация хеш-таблицы, вынесенная в отдельный переиспользуемый модуль:

```c
hashmap_t *hashmap_create(void);
void      *hashmap_get(const hashmap_t *map, const char *key);
int        hashmap_set(hashmap_t *map, const char *key, void *value);
void       hashmap_delete(hashmap_t *map, const char *key, void (*free_value)(void *value));
bool       hashmap_for_each(hashmap_t *map, hashmap_callback_t callback, void *ctx);
void       hashmap_destroy(hashmap_t *map, void (*free_value)(void *value));
```

- 256 бакетов, коллизии разрешаются цепочками
- Хеш-функция — FNV-1a по байтам строки
- Значение хранится как `void *`, освобождение — на стороне вызывающего кода (через переданную функцию `free_value`), модуль не привязан к специфике KVS

---

### 5. Logger

Записывает лог-сообщения с маркерами `Info` / `Error` / `Warning` и временной меткой в файл.

## 🧪 Тестовые сценарии

`main.c` проверяет:

1. **Happy flow** — запись, чтение, обновление, удаление, чтение несуществующего ключа
2. **Значение больше страницы** — запись и чтение значения > 512 байт (многостраничная запись)
3. **Persistence** — закрыть и снова открыть хранилище, убедиться что данные сохранились
4. **Потеря питания** — симуляция обрыва записи на полуслове, проверка что неполная запись не читается
5. **CRC** — внешнее искажение байта в файле, проверка что CRC ловит повреждение
6. **GC** — заполнение flash до срабатывания Garbage Collection, проверка что данные доступны после уплотнения

Результаты пишутся в лог-файл с маркерами `Info` / `Error`.
